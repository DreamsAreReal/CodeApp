# Ревью — ЛИНЗА: Робастность и edge-cases

Оценщик: внешний скептик. Метод — исполняемо, живой продукт. Собрал текущий фронт
(`VITE_API_BASE=` → same-origin), поднял backend (dotnet 10, :5080), сшил их
same-origin через прокси (:4190), прогнал Playwright-харнесс по неидеальным путям.
Скриншоты — `docs/reviews/evidence/edge/*.png`. Харнессы — `.verify/_review-*.mjs`.

## ВЕРДИКТ — ДОРАБОТАТЬ (не блокер релиза, но 2 заметных дефекта робастности)

Ядро надёжно: сборка зелёная, офлайн/сеть на всех трёх верхних экранах даёт общий
дружелюбный error-card + рабочий retry + восстановление, двойной тап по оценке шлёт
ровно 1 POST, пустой ввод не сабмитится, длинные строки не ломают вёрстку, 320px
держит. Но два реальных пробоя в неидеальных условиях + один визуальный конфуз.

**Паспорт вкуса: PASS** (доказательство: `evidence/edge/cur-home-390.png`,
`cur-home-320.png`, `cur-lesson.png` — крем #F6F1E9 + коралл #F0533A + Rubik,
spark-знак в топбаре присутствует и заметен, слоп-запреты соблюдены). ПРИМЕЧАНИЕ:
приложенные к заданию `docs/evidence/product-readiness/*.png` СТАРЫЕ (топбар в них
показывает текст «Фундамент C#» + auth через `?userId=` — это добилдовая версия из
wwwroot). Текущий исходник рендерит только spark-знак; критиковал ТЕКУЩИЙ код по своим
свежим снимкам.

## НАХОДКИ

### P1 — Тумблер «Меньше движения» НЕ выключает анимации уроков
Локация: `app/src/engine/dom.ts:7-12` (`prefersReducedMotion`) vs
`app/src/app/settings.ts` (`codex.reducedMotion` → класс `.reduced-motion`) ·
`app/src/app/lessonRunner.ts:22` (`const REDUCED = prefersReducedMotion()`).

Нарушенное требование: «reduced-motion — всё ли уважает?». Единственный
пользовательский регулятор движения в продукте (Профиль → «Меньше движения», хинт
«Отключить анимации и переходы») НЕ отключает самые заметные анимации — живые
диаграммы уроков.

Причина: движок читает `prefersReducedMotion()` = только OS-медиазапрос
`prefers-reduced-motion` + `?reduced=1`. Он НЕ читает localStorage-флаг, который
пишет тумблер. Класс `.reduced-motion` глушит только CSS `animation/transition`
(`progress.css:351`), а диаграммы анимируются через WAAPI (`element.animate`) с
явными JS-длительностями + autoplay — CSS-правило на них не влияет.

Доказательство (исполнено, `.verify/_review-rm-frames.mjs`):
```
[toggle-ON  os-OFF] engineReduced=false  distinctFrames=3/5   ← анимация идёт
[os-ON toggle-OFF]  engineReduced=true   distinctFrames=2/5   ← OS уважается
```
Плюс `.verify/_review-robustness-edge.mjs` → `reducedMotion.LESSON_engine_reducedFlag=false`
при `localStorage=1` и `htmlHasClass=true`; и после reload то же самое.

Кого бьёт: пользователи Telegram на десктопе/Android без OS-настройки reduce-motion —
для них тумблер обещает и не выполняет (доступность / вестибулярный триггер).

Фикс: `prefersReducedMotion()` должен ИЛИ-ить в себя `reducedMotionEnabled()` из
settings.ts (OS-медиа || URL || localStorage-флаг). Тогда и модульный `REDUCED`, и
`VizPlayer.cfg.reducedMotion` подхватят тумблер. Мелочь: `REDUCED` вычисляется на
загрузке модуля — после смены тумблера нужен ре-эвал при входе в урок (сейчас урок
рендерится заново при каждом `showLesson`, так что чтения флага в момент входа
достаточно).

### P1 — Гонка «устаревший рендер затирает экран»: медленная навигация уводит с текущего экрана
Локация: `app/src/app/router.ts:39-69` (`showHome/showProgress/showProfile` — нет
guard-а поколения) · экраны пишут `root.innerHTML` по резолву своего fetch.

Нарушенное требование: «двойные тапы, гонки». Нет защиты от конкурентных асинхронных
рендеров: каждый `render*` пишет в общий `#app` когда его запрос вернулся, не проверяя,
не ушёл ли пользователь дальше.

Сценарий (реально воспроизведён): пользователь тапает «Прогресс» (медленная сеть),
затем быстро «Профиль». Профиль грузится и оседает — пользователь смотрит Профиль.
Через ~1.5с доезжает медленный `/api/progress` → поздний `renderProgress` затирает
`innerHTML` и экран САМ перепрыгивает обратно на Прогресс.

Доказательство (`.verify/_review-navrace.mjs`):
```
MIDWAY (сразу после тапа Профиль): activeTab=profile, title="Профиль", hasProfile=true
AFTER  (медленный progress доехал): activeTab=progress, title="Прогресс", hasProfile=false
```
Скрин: `evidence/edge/06-navrace-clobber.png`. Также в общем харнессе `navRace.tabMatchesBody=false`.

Фикс: ввести монотонный токен навигации в Router (`this.nav++` на каждом `show*`);
`render*` после await сверяет свой токен с текущим и НЕ пишет в DOM, если устарел.
Либо AbortController на fetch при смене экрана.

### P1 — Ошибка сохранения оценки: сырой техтекст + галочка «успех» вместо дружелюбной ошибки
Локация: `app/src/app/lessonRunner.ts:795-801` (`wireGrade.post` catch).

Нарушенное требование: «error+retry покрывает ВСЕ вызовы?» + консистентность. Это
ЕДИНСТВЕННЫЙ сетевой вызов в петле повтора, который НЕ проходит через общий error-card.
При падении `POST /api/review` пользователю показывается сырая строка
`«Ошибка сохранения: network: Failed to fetch»` — техническая англ-вставка в русском
UI — И РЯДОМ ЗЕЛЁНАЯ ГАЛОЧКА (контейнер `#gradeDone` всегда рисует check-иконку). То
есть проваленное сохранение выглядит как успех.

Доказательство: `.verify/_review-robustness-edge.mjs` → `reviewFail.usesSharedErrorCard=false`,
`gradeMsgShown="Ошибка сохранения: network: Failed to fetch"`. Визуал:
`evidence/edge/07-review-fail-raw.png` (галочка ✓ над сырой ошибкой).

Смягчающее: кнопки оценок ре-активируются (`sent=false`), повтор тапа при вернувшейся
сети сохраняет (`recoveredOnRetry=true`) — поэтому P1, не P0. Но это самый ценный экран
(момент, когда двигается FSRS-расписание), и он единственный ломает паттерн.

Фикс: в catch — убрать/спрятать check-иконку `#gradeDone`, показать строку из
strings.ts (напр. переиспользовать `S.resetError`-подобный дружелюбный текст +
«Повторить»), не пробрасывать `e.message`. Техдеталь — максимум муто-мелким, как в
общем error-card.

### P2 — Тупик при неизвестном lessonId (нет nav / retry)
Локация: `app/src/app/lessonRunner.ts:137-141`.

`runLesson` при `getLesson(id)===undefined` рисует `.notice err` + кнопку «На главную»,
но БЕЗ нижней навигации. Кнопка спасает, так что не тупик де-факто, и id приходят из
реестра (маловероятно). Но паттерн отличается от общего error-card. Низкий приоритет.

Фикс: обернуть в общий frame с navBar, либо переиспользовать `errorCard`.

### P2 — Сырой англ-детейл `network: Failed to fetch` на основном error-card
Локация: `app/src/app/ui.ts:37` (`err-detail`), виден на всех offline-экранах.

Строка техническая и англоязычная в консюмерском RU-UI. Она намеренно приглушена и
мелкая (`evidence/edge/03-offline-home.png`) — «dev afterthought», консистентна везде,
так что терпимо. Но для планки «готовый продукт» стоит либо локализовать частые случаи
(«Проверьте интернет»), либо прятать техдеталь за тап «подробнее».

### P2 — Console.error на офлайне (браузерный, не апповый)
`net::ERR_INTERNET_DISCONNECTED` в консоли при офлайне — это браузер логирует
неудавшийся ресурс, не всплытие апповой ошибки. Приложение ловит и показывает card.
Строго — консоль не «чистая» под сетевым сбоем, но апп тут ни при чём. К сведению.

## ЧТО ПЛОХО (обязательная секция)
1. Тумблер reduced-motion не глушит анимации уроков — обещание не выполняется (P1).
2. Медленная навигация перебивает более поздний выбор экрана — экран «прыгает» сам (P1).
3. Провал сохранения оценки: сырой англ-текст + галочка успеха на ошибке (P1).
4. Сырой `network: Failed to fetch` на всех error-card — англ в RU-UI (P2).
5. Приложенные evidence-скрины СТАРЫЕ (топбар с текстом, `?userId=` auth) — не
   соответствуют текущему коду; продукт-паспорт по ним ввёл бы в заблуждение.

## ЧТО ХОРОШО (кратко)
- Офлайн/сеть на home/progress/profile: общий error-card + retry + восстановление,
  без сырых утечек в основной поток, nav не пропадает (не заперт).
- Двойной тап по оценке → ровно 1 POST; пустой ввод не сабмитится; двойной «Проверить»
  → 1 баннер. Гонки в самом уроке закрыты флагами `answered/sent`.
- Длинные строки/переполнение: топик-сабтайтлы обрезаются эллипсисом, топбар-статы
  тянутся, `#conn` переносится в рамке — горизонтального скролла нет (scrollW==clientW).
- 320px держит без слома; 3 хиро-состояния (session/done/empty) рендерятся корректно.
- `?reduced=1` и OS-медиа reduce-motion РАБОТАЮТ (движок статичен) — база уважения есть,
  не хватает только связки с тумблером.
- FSRS-петля реальна: `POST /api/review` двигает расписание server-side (200, не display-only).

## ПРОВЕРКА ДОКАЗАТЕЛЬСТВ
Не доверял ни progress-файлам, ни приложенным PNG. Перепрогнал: build (exit 0),
backend (dotnet run, :5080 200), same-origin прокси, 6 Playwright-сценариев + 3
прицельных. Все числа — из исполненного кода (см. вывод харнессов выше). Обнаружил,
что артефакт в `backend/wwwroot` — устаревшая сборка (401 против текущего backend,
IDOR-паттерн `?userId=`), поэтому ревьюил свежесобранный текущий исходник.

## СОМНЕНИЯ
- Внутри реального Telegram-клиента (не браузер) поведение native BackButton/свайпов

- Хаптику/звук не проверял (каналов нет) — вне линзы.
