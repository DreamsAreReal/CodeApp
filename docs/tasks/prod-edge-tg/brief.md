# ТЗ — Прод-вход для Telegram Mini App (CodeApp)

## Цель
Пользователь Telegram открывает бота @coseappbot → кнопка меню открывает Mini App по
постоянному HTTPS-домену → приложение работает; вход переживает редеплой, перезагрузку
VPS и продление сертификата (доказано замерами снаружи).

## Контекст
Приложение задеплоено (контейнер codex, 127.0.0.1:8091, healthy; CI зелёный), но
публичного входа нет. Кнопка меню бота уже существует («Учить C#») и указывает на
cloudflared quick-tunnel, который 17.07 сменил URL — кнопка мертва, пользователи видят
белый экран. Инсайты: RS-00 (углы 1–5), RS-01 (домена/сертов на VPS нет; 443 закрыт
только ufw; порт 8091 стабилен в /opt/codex; бот на long polling), RS-02 (TG требует
HTTPS с валидным CA-сертом, IP нельзя; setChatMenuButton достаточно; WebView →
анти-frame-заголовки не мешают).

## Класс результата
software-service/бот, подкласс «инфраструктура/прод-вход» (improvement). Класс вне
реестра профилей → секция «Выведенные гейты класса» ниже; финальную таблицу сверяет critic.

## Целевая планка
ГОТОВЫЙ ПРОДУКТ (ступень ≥3): вход надёжен (автопродление серта доказано dry-run-ом,
порт стабилен при редеплое, HTTP→HTTPS редирект), а не «однажды открылось».
Цена планки: ~1 майлстоун builder-а (оценка; сложных фичей нет, риски — внешние).

## Скоуп
### P0 (без этого задача провалена)
- Домен с A-записью на 192.3.94.42 (источник — решение гейта).
- ufw allow 443/tcp; nginx server-блок: HTTPS → proxy_pass 127.0.0.1:8091, HTTP→HTTPS редирект.
- Let's Encrypt серт (HTTP-01 по умолчанию; спорную рекомендацию DNS-01 из RS-02 проверяем фактом) + автопродление, доказанное `certbot renew --dry-run`.
- Кнопка меню бота обновлена через Bot API setChatMenuButton на https://<домен>/ с СОХРАНЕНИЕМ текста «Учить C#».
- Соседние сервисы VPS (xray, zhaba, portfolio-new, concert-vote, deploy-hub-demo, VPN-nginx) не задеты — доказано до/после.
### P1 (важно)
- Вывод codex-tunnel.service (stop+disable) ПОСЛЕ верификации домена — туннель больше не источник правды.
- Обновить deploy/Caddyfile.example → фактическую доку прод-входа в репо (nginx-вариант).
### P2 (желательно)
- Внешний uptime-мониторинг (например, бесплатный пинг-сервис) на https://<домен>/health/live.

## Не входит в скоуп
- Изменения кода приложения, CI, deploy-hub.
- Direct Link Mini App (t.me/bot/appname — требует BotFather вручную).
- Миграция соседних сервисов на домен/443.

## Полномочия
- Разрешено (запрашивается на гейте): `ufw allow 443/tcp`; установка certbot; создание
  ОТДЕЛЬНОГО nginx server-блока + reload после `nginx -t`; выпуск LE-серта; запись
  кнопки меню setChatMenuButton (с сохранением текста); после верификации — stop+disable
  codex-tunnel.service. Всё обратимо (бэкапы, откат правила/кнопки).
- Лимиты: чужие конфиги (xray, VPN-nginx блок, соседние compose) НЕ трогать; деньги НЕ тратить.

## Ограничения
VPS общий; nginx уже занят под VPN-инструкцию — только новый server-блок. Стек: nginx +
certbot (Caddy не ставим — второй edge-сервер на общем боксе = конфликт за 80/443).

## Риски и допущения
- RS-02 vs RS-01: «DuckDNS требует DNS-01» — спорно; допущение: HTTP-01 сработает
  (A-запись есть, 80 открыт); если нет — фолбэк DNS-01 c плагином, +15 мин.
- setChatMenuButton может отличаться от «кнопки по умолчанию» BotFather → проверяем
  getChatMenuButton после записи (G4).
- Допущение: пользователь предоставит домен/аккаунт на гейте; иначе времянка —
  обновить кнопку на живой tunnel-URL (честно помечено как НЕ-DONE, ступень 1).
