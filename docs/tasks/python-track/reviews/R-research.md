## 2. ГЛУБИНА (вопрос 1): не первая страница гугла — подтверждаю

- RS-01 несёт реальные senior-нюансы, а не оглавление md: `__defaults__` у объекта
  функции; поверхностная неизменяемость tuple; def-time исполнение декоратора;
  «декоратор съел результат» (c07); `return` в finally, переписывающий return и
  глотающий исключение (c12); теневой instance-атрибут (c15); name mangling как
  «не приватность»; одноразовость genexpr (c09); «await подряд не ускоряет» с
  реальным замером c24. Это вопросы, на которых валятся мидлы, — планка сеньора.
- RS-02 добавляет слой «как будто сам спроектировал язык»: GIL, refcount+циклический
  GC, generator frame (locals+instruction pointer — дословная цитата Reference),
  closure cells, дескрипторный протокол как единый механизм bound method/property/
  classmethod, small-int cache. Это НЕ пересказ md — md этого не содержит (проверено
  по инвентарю RS-01: слова «GIL» в md нет).
- Уникальная ось продукта (мостики C#↔Python, §6 RS-01) деконструирована до уровня
  переиспользуемых сцен (`[2,2,2]` ↔ «333» из closures.ts) — это фирменный крючок,
  которого нет ни у Python Tutor, ни у Real Python (угол 12 RS-00).

Вывод: глубина принята. Но см. must-fix 3 — один из «вау»-гэпов сам содержит
потенциальную неточность.

- URL-банк: 32 URL — точные страницы с #якорями (exception-hierarchy,

## 4. ПРОТИВОРЕЧИЯ между отчётами (вопрос 3)

1. **run-python endpoint.** RS-01 §10: «Перед первым уроком — добавить `run-python`
   в бэкенд»; RS-00c риск №5: три варианта (A endpoint / B manual-verify / C MCQ)
   с требованием решения на DESIGN; STATE (решение оркестратора): «expect считать
   локальным python3, run-csharp только для C#». Три документа — три позиции.
   Плюс режим задачи improvement, бэкенд-движок LOCKED: новый endpoint = молчаливое
   расширение скоупа. → must-fix 2 (одно решение, зафиксированное в ТЗ как ADR).
2. **Версия Python.** RS-01/RS-02 считали на 3.9.6; в контуре есть python3.12;
   md местами учит 3.10+ синтаксису. Карточка c25 (py-type-hints, сегмент 2) на
   3.12 МЕНЯЕТ СМЫСЛ: `int | None` больше не падает, expect «TypeError» станет
   ложью, сегмент «gate по версии» надо переворачивать (правильный урок для
   раннера-3.12: «работает с 3.10, падало на 3.9»). → must-fix 1 (must-check-список ниже).
3. **RS-01 не знает об ошибках md, найденных RS-02.** P1-3 (py-exceptions) берёт
   «иерархию BaseException» из §11 md — а RS-02 Ошибка 3 доказала, что схема md
   НЕВЕРНА (FileNotFoundError/ConnectionError — дети OSError, не Exception).
   P1-1 сегмент 2 формулирует «без nonlocal — новая локальная» — та же
   упрощёнка, что Ошибка 1 RS-02 (реально UnboundLocalError на компиляции, gap
   B-12). Урок «объектная модель» включает «пример полиморфизма выучить дословно»
   — Ошибка 2 (код падает). Без явной склейки нарезка RS-01 протащит ошибки md
   в карточки. → must-fix 4 (merge-карта).
4. **RS-00b vs LOCKED.** Ставки 1 (live-песочница), 5 (динамический FSRS), 8
   (Pydantic real-time) требуют новой механики движка/FSRS — прямо запрещено
   рамкой improvement (движок/дизайн/формат LOCKED). Брейншторму можно мечтать,
   но ТЗ обязано явно отклонить их с ссылкой на рамку, иначе scope creep.
   Дополнительно RS-00b содержит фактические небрежности («в Python нет boxing» —
   в CPython всё как раз «boxed»; «замыкание = асинхронная функция помнит контекст
   task'а»; адреса-«ячейки» 0x7fff) — материал НЕ для переноса в ТЗ без фильтра.
5. Мелкое: RS-01 «~30 MB» (md) vs RS-02 «40.3 MB» — разрешено в пользу замера,
   но цифра платформо-/версиезависима: в карточку/сегмент — только «десятки MB».

## 5. ПОЛНОТА для ТЗ (вопрос 4): чего не хватает конкретно

- **Merge-карта RS-01×RS-02** (must-fix 4): таблица «урок → какие фиксы A-1…A-9 и
  гэпы B-1…B-6/B-12 в него вплетаются». Сейчас RS-02 адресует md-разделы (§12),
  RS-01 — слаги уроков; транзитивная склейка в голове читателя = ошибки потеряются.
- **Решение по версии раннера** (must-fix 1): G-EXEC-PY сформулирован как «версия
  зафиксирована», но КАКАЯ — не решено. Это вход ТЗ, не DESIGN: от версии зависят
  expect всех 28 карточек, опкоды dis-панели и смысл сегмента P0-6.2.
- **Чеклист «допустимый expect»** (must-fix 5): RS-01 разрозненно упоминает
  (set не печатать, тайминги не брать, stdout≠stderr) — собрать в один авторский
  гейт-чеклист детерминизма, иначе G-EXEC-PY проходится Гудхартом (см. §6).
- **Лимит MCQ-fallback**: P0-6 c2 и P0-7 c2 уже уехали в MCQ; RS-00c сам пишет —
  «узнавание, не генерация, нарушает методику». Нужен порог в ТЗ (например,
  ≤1 MCQ на урок и только там, где exec-вывод недетерминирован).
- **G6/пейсинг для Python** (находка 9 RS-00c) — никем не закрыта: ни baseline,
  ни явного «наследуем C#-калибровку как гипотезу и меряем после волны». Нужна
  явная строка в ТЗ.
- **FSRS-поток новых карточек** (находка 6 RS-00c): P0 = 19 карточек (не 60, страх
  преувеличен — RS-01 это фактически снял), но правило ввода в очередь (дозирование
  по урокам/дням) в ТЗ всё равно нужно.
- **P2 «cheatsheet-экран без анимаций»** (RS-01 §4-P2): это НОВЫЙ формат экрана —
  конфликт с LOCKED «формат». Либо P2 = обычные карточки в существующем формате,
  либо явный ADR на гейте. Сейчас — мина.
- Открытые мелочи, честно помеченные ресерчем (track id, pydantic в раннере,
  цитаты на каждый claim при сборке) — в ТЗ как входы DESIGN/BUILD, ок.

## 6. Гудхарт (вопрос 5)

- **Реестр покрытия «17/17, 10/10»**: метрика «размечено» проходима разметкой
  чего угодно в P2. Фактическая нарезка value-driven (P0 = механизмы, справочники
  честно выкинуты из уроков — угол 3 RS-00 применён), так что здесь Гудхарт не
  сработал. Принято.
- **G-EXEC-PY проходим без ценности**: карточка может иметь реальный stdout и
  быть при этом (а) недетерминированной на раннере (set-порядок, хэш-рандомизация,
  тайминги), (б) версиозависимой (c25, тексты ошибок UnboundLocalError 3.9 vs
  3.11+), (в) педагогически ложной (см. small-int ниже). Гейт нужно доопределить:
  expect детерминирован + stdout-only + version-pinned. → must-fix 5.
- **«Сеньор понял механизм» vs «покрыть md»**: сегменты типа P0-5.7 (chip-маппинг
  дюндеров) и P2-колоды — зона риска «список фактов вместо сюжета» (риск №2
  RS-00c). На DESIGN каждому сегменту — тест «есть ли у сегмента процесс/сюжет»;
  в ТЗ достаточно одной строки-гейта.

## 7. Отдельная находка точности: B-6 small-int cache (must-fix 3)

RS-02 рекомендует в волну 1 спайк «`256 is 256 → True`, `257 is 257 → False`
(в разных statement)». В отличие от раздела A, для B-6 исполненный вывод НЕ
приложен, и утверждение в таком виде опасно: в CPython литералы в ОДНОМ модуле/
code object складываются в co_consts и часто интернируются → в скрипте
`a = 257; b = 257; print(a is b)` типично печатает True; классическое False
воспроизводится в REPL или через разные единицы компиляции, а с 3.8 литеральный
`x is 257` ещё и SyntaxWarning. Т.е. поведение implementation-defined и зависит
от способа запуска — ровно то, что НЕЛЬЗЯ класть в predict-output. Это
рекомендованный «вау»-гэп волны 1, содержащий потенциальную фактическую ошибку —
прямое попадание в риск №1. Требование: реальный прогон на версии раннера в том
виде, в каком карточка будет исполняться (файл, не REPL), и либо переформулировка
сегмента («identity малых int — деталь реализации, поэтому `is` для чисел — баг»),
либо отказ от exec-карточки по этой теме.

## 8. ЧТО ПЛОХО (сводно, ≥3 — здесь 7)

1. Вся доказательная база на 3.9.6 при целевом контуре 3.12; «стабильно по 3.8–3.12»
   — недоказанное утверждение (нарушение правила доказательств).
2. Трёхстороннее противоречие по run-python (RS-01 vs RS-00c vs STATE) не разрешено
   и толкает к расширению LOCKED-скоупа.
3. B-6 (small-int) — рекомендация волны 1 без исполненного артефакта, с вероятной
   ошибкой формулировки (риск №1 продукта).
4. Нет merge-карты «уроки RS-01 × фиксы/гэпы RS-02» — найденные ошибки md могут
   не доехать до карточек.
5. G-EXEC-PY без чеклиста детерминизма — проходим Гудхартом; MCQ-fallback без лимита.
6. P2 «cheatsheet-экран без анимаций» — тихий новый формат против LOCKED.
7. RS-00b содержит фактические небрежности и анти-LOCKED ставки без маркировки
   «отклонено рамкой» (само по себе не блокер — брейншторм, — но фильтр в ТЗ обязателен).

2. **ADR по механизму verify Python-карточек.** Один абзац: выбранный вариант

   с явным снятием LOCKED на гейте), остальные — отклонены. Проверка: RS-01 §10 и
   STATE ссылаются на один и тот же ADR, противоречие исчезло.

4. **Merge-карта**: таблица «урок → A-фиксы/B-гэпы RS-02» (минимум: A-1+B-12 →
