## ТОП-5 РИСКОВ (с последствиями и как проверить)

### РИСК №1: Конфликт аудиторий разрушает весь контент
**Наблюдение:** md-файл ориентирован на AQA-собеседования («на собесе спросят»), но приложение — для СЕНЬОРА глубиной технологий. Тон, акценты, примеры не совпадают.

**Последствие:** Уроки либо остаются поверхностно-справочными (не лезут в машинный уровень, как требует app), либо перерабатываются половину времени разработки. Сеньор закроет приложение на slide-3 (скучно), джун не откроет вообще (слишком глубоко).

**Как проверить:** Прямо на гейте TЗ: перепроверить 5 разделов md на тон/глубину против эталона lesson-boxing (эталон = 7 анимирован. разборов, машинный уровень, сеньор-нюансы). Если разрыв >30% — БЛОКЕР. Калибровка: переписать акценты (убрать «шпаргалка для собеса», добавить IL/bytecode/how-why).

### РИСК №2: Справочные разделы мертвы в приложении
**Наблюдение:** 15 разделов md включают «Стандартная библиотека для тестов» (§13) и «Common idioms» (§15). Это — справочники. Их НЕВОЗМОЖНО анимировать: нет сюжета, нечего визуализировать, 20 фактов в список.

**Последствие:** Уроки превращаются в просмотр списка текста + несколько иконок. FSRS-карточки = скучные MCQ-fallback. Волна не вмещается в объём, но вычёт справочников оставляет дырей (юзер переходит на гугл). Приложение перестаёт быть самодостаточным.

**Как проверить:** На фазе RESEARCH (сейчас): обязательный аудит каждого из 15 разделов — есть ли у раздела СЮЖЕТ (последовательность понимания)? Справочники (§13, §15, части §3, §11) → отметить как P2/адаптировать/вырезать. Не ждать фазы BUILD.

### РИСК №3: Точность md-упрощений попадает в карточки
**Наблюдение:** 00-python-refresher.md — это КОНСПЕКТ для людей (с упрощениями типа «dict = пара ключ-значение»). Приложение требует АБСОЛЮТНОЙ точности на уровне stdlib/docs.python.org.

**Последствие:** Карточка попросит предсказать вывод, ответ будет неверен в боевом Python из-за упрощения в md. Пример: слайд про f-strings говорит «подставляет выражение», но не уточняет order-of-evaluation, escaping, рекурсию форматирования. Юзер отвечает неверно, теряет очко FSRS. Вторая волна уже содержит ошибку в знании (Goodhart-тест пройден вручную, не логикой).

**Как проверить:** Гейт авторинга G-ACCURACY-PY (новый, аналог G2 для C#): перепроверить 100% фактов-утверждений в каждом уроке Python (не просто от пользователя — ПРОВЕРИТЬ ВЖИВУЮ):
- Запустить `python3` локально для каждого примера кода.
- Сверить цифры/параметры с docs.python.org и PEPs.
- Если факт упрощён → добавить сегмент nuance (объяснить, где упрощение и почему).

### РИСК №4: Объём 15 разделов не влезет в волну-1
**Наблюдение:** Briefs указывает планку: волна 1 = ~10–12 уроков (3–4 флагмана по 7 сегм. + 6–8 стандартных по 2–3). Эталон boxing = 7 разборов = ~7–8 часов работы. 15 разделов × среднее 4 сегм. = 60+ карточек.

**Последствие:** (a) если все 15 вместить → волна разбухает, сроки срываются, качество рушится. (b) если выбрать «Блок 0 / 11» (как в md-заголовке) → DUNDERLEY, в приложении нет контекста блоков 1–11. (c) если нарезать только топ-6 — дыра в foundation (не поймёшь async/await без генераторов, не поймёшь декораторы без замыканий).

**Как проверить:** ДИЗАЙН-гейт: выписать все 15 разделов в таблицу (название | # карточек | сложность | зависимости). Провести иерархию-дерево зависимостей. Выбрать один из сценариев: (A) 6–8 ОБЯЗАТЕЛЬНЫХ основ (синтаксис, типы, коллекции, функции, коротко декораторы-генераторы-с-with-классы, errors + async + type-hints) = ~30–35 карточек = реальная волна-1. (B) Все 15, но P2 режим = full curriculum, после волны-1 расширяем. ЗАБЛОКИРОВАТЬ БЕЗ ЯВНОГО ВЫБОРА.

### РИСК №5: Python-карточки не могут использовать существующий run-csharp
**Наблюдение:** Приложение имеет бэк-эндпоинт `/api/authoring/run-csharp` (запускает dotnet-код, возвращает stdout). Для Python-карточек этот эндпоинт бесполезен. Хороший ответ carточки = реальный вывод `python3`, но вычислять его ОТКУДА?

**Последствие:** 
- (a) Ответы карточек predict-output ДОЛЖНЫ быть собраны вручную (python3 локально для каждой) и заморожены в `verify.expect`. Если расходимость — карточка показывает НЕ то, что ожидает.
- (b) Если БЕЗ `verify.expect` → fallback на MCQ (выбери из 4 вариантов). Это — узнавание, не ГЕНЕРАЦИЯ, нарушает методику (RS-09 требует generation для сеньора).
- (c) «Авторский гейт» G-EXEC-PY: каждое predict-output должно пройти локальный run python3 → зафиксировать stdout. Это — N дополнительных операций за урок, не автоматизируемо.

**Как проверить:** На гейте DESIGN: принять решение: (A) собрать run-python эндпоинт на бэке (Python subprocess sandbox); (B) принять, что Python-карточки = predict + manual-verify (документировать граблю); (C) переход на MCQ-fallback для Python. БЕЗ решения — гейт не пройти. Сейчас это пустой вакуум в спеке.

## ЕЩЕ 4 НАХОДКИ (неловкие темы + технические дыры)

### Находка 6: FSRS-очередь может быть завалена карточками Python
Приложение = shared FSRS-очередь (все треки). Добавляем 60+ карточек Python → первый день юзер получает 20 карточек вместо 3–5 (привычный паттерн C#). Это НАРУШАЕТ психологию daily-loop (Goodhart: юзер пропускает дни, потому что слишком много).

**Как закрыть:** DESIGN-гейт: обязателен вторичный параметр за пользователем = «какие треки включены?». Если Python выбран впервые, он СТАРТУЕТ С МАЛОГО (первый день = 3 карточки Python, C# в фоне). Или: волны Python асинхронны разросту (не все 15 разделов волной, распределить на 3–4 волны по 4–5 разделов).

### Находка 7: Техдыра — подсветка синтаксиса для Python
Приложение встраивает Shiki.js для C#-snipetов (`kind: "code"` в segments). Для Python шрифт код = JetBrains Mono (OK), но **НЕТ конфига Shiki для Python** в приложении (только C#). Если вставить питоновский код — подсветка упадёт (весь код чёрным, потеряется читаемость).

**Как закрыть:** INFRA-гейт: добавить в Shiki-конфиг язык Python (тривиально, 1 строка). Тест: собрать урок с python-снипетом, проверить подсветка играет (ключевые слова, literals, comments).

### Находка 8: Запуск примеров на il-панели (IL для Python)
Эталон boxing использует IL-панель (`scene.ilLine`) — показывает IL-код параллельно анимации памяти. Это ФИШКА для C#/.NET.

Для Python IL === **bytecode (dis module)**. Но:
- (a) Вывод `dis.dis()` — это ТЕКСТ (много строк), не визуальная сцена.
- (b) CPython bytecode vs PyPy vs Cython = разные картинки.
- (c) Сеньор Python → ориентирован на CPython, но нюансы GC/ref-count НЕ видны в bytecode.

**Следствие:** Уроки Python получат ДРУГУЮ точку фокуса (не IL, а ref-count + GC-счётчики?). Это меняет design-язык урока против C#-эталона → нарушение унификации.

**Как закрыть:** На DESIGN-гейте: решить, как будет выглядеть аналог IL-панели для Python. Варианты: (A) dis-output как текстовый, (B) абстрактная VM-сцена (ref-count, GC), (C) не используем (Python-сегменты тип. другие).

### Находка 9: Конфликт гейтов - FSRS-калибровка vs Python-сложность
Бриф указывает гейт G6 (Пейсинг): полоса успеха сессии = 83–88% (медиана N прогонов). Это отрегулировано на C#-уроках.

**Проблема:** Python-уроки на новичков (даже сеньора) могут быть ПОСЛОЖНЕЕ (более абстрактные концепты: yield vs async, декораторы vs context managers, замыкания), или ПРОЩЕ (простая синтаксис). Калибровка сложности карточек ПОТРЕБУЕТ переподстройки сим-харнесса на новой когорте (новый baseline).

**Как закрыть:** RESEARCH-гейт: собрать 10-15 реальных примеров карточек, прогнать на тестовой когорте (даже вручную), вывести новый baseline успеха для Python. Если он далёк от 83–88%, обновить G6 или перетарировать сложность карточек.
