## Топ-5 Максималистских Ставок

### 1. **Live Python Execution в карточке** (сигнатура-в-скелете)
**Идея:** каждая ключевая концепция раскладывается в **интерактивную песочницу прямо в карточке** (не отдельное видео), где пользователь может менять код и видеть результат за ~50мс. Пример: карточка про `yield` → слайд 1: генератор «замораживается на yield»; пользователь нажимает «next()» → выполняется дальше по коду; слайд 3: тест-фикстура с той же мехиникой, но в pytest-контексте.

**Почему вау:** Никто не показывает асинхронное выполнение **в микродозах**. У конкурентов видео-разбор или текст на слайде. Здесь — ты ЧУВСТВУЕШЬ паузу, управляешь ею, видишь состояние памяти (локальные переменные) между `yield`-ами. Это **cognitive lock-in** — раз почувствовал, забыть не можешь.

**Цена:** реестр потребления → первый пользователь вкусит суперсилу за 2 минуты вместо 15; повтор (FSRS) = укреплён в мышце.

### 2. **Таблица Мутации Памяти: Immutable vs Mutable как Movie** (мост C# ↔ Python, анимация)
**Идея:** раздел про mutable/immutable (§1.3) — не скучные диаграммы, а **пошаговая анимация памяти в реальном времени**. Пример:
- Слайд: `x = [1, 2, 3]` → ячейка памяти на адресе `0x7fff` с содержимым.
- Слайд: `lst[0] = 99` → содержимое той же ячейки меняется (стрелка мигает).
- Параллельно: `s = "hello"` → новая ячейка `0x8000`.
- Слайд: `s = "Hello"` → **новая** ячейка `0x8004`, старая `0x8000` умирает.

Связь с C#: зарисовка «reference type (class) ↔ value type (struct)» → Python `list`/`dict` = reference type → мутируют в памяти; `str`/`tuple` = value type → новые объекты.

**Почему вау:** Senior-разговор на собесе — «что волкло с mutable default в функции» — встаёт в голове как кино, не как текст. Это ровно тот чит-код на `def add(items=[])`, который **один раз увидел в памяти — вспомнится на собесе 100%**.

**Цена:** ошибки в коде (shared state) исчезают; ревью-комментарий «это mutable» → сразу видно почему.

### 3. **Собес-Супермен: Трансляция C#-вопроса в Python** (межъязыковой коридор)
**Идея:** слайд-последовательность **вопрос C# на собесе → как его переформулировать в Python → как ответить используя Блок 0**. Примеры:
1. **C#:** «Разница boxing и unboxing?» → **Python:** «А как в Python работает передача примитивов в функцию?» → **Разбор:** Python dinamicamente typizirован, нет boxing (type как часть объекта), но `None` меняет тип рефа; заводская практика — `isinstance(x, (int, float))`.
2. **C#:** «Что такое замыкание?** → **Python:** `make_multiplier(factor)` → как `factor` помнится в `multiply` → это ровно то же, что **асинхронная функция помнит контекст task'а**.
3. **C#:** «Как работает `await`?» → **Python:** «Это как `yield` — пауза и возврат управления, но диспетчеру event loop'а, не caller'у».

**Почему вау:** Мозг уже знает C# на уровне синапсов. Трансляция → моментальная фиксация в Python по ассоциации. Это **мост через известное территории**.

**Цена:** на AQA-собесе сказал на C# → сразу контрпример на Python → impression «знает оба языка и видит общие паттерны» = Staff-уровень.

### 4. **Декоратор-Йо: Interactive Trace через несколько слоёв** (сигнатура-в-скелете 2)
**Идея:** карточка про декораторы (§6) — **интерактивная пошаговая трассировка вызова** для `@repeat(3) → @measure_time → def greet()`. Пользователь жмёт «шаг» и видит:
- Шаг 1: `repeat(3)` вызвана → возвращает `decorator(func)`.
- Шаг 2: `decorator(greet)` вызвана → возвращает `wrapper`.
- Шаг 3: `greet = wrapper` → перепривязка имени.
- Шаг 4: вызов `greet()` → `wrapper` выполняется.
- Слайд 5–7: трассировка **внутри** `wrapper` для трёх повторений.

Соч слайд: **асинхронная декорация** (в pytest-контексте из Блока 1: `@pytest.fixture(scope="session", autouse=True)` — 3-слойная фабрика).

**Почему вау:** Junior на собесе: «Не может объяснить как работает `@pytest.fixture(scope='session')`» → пошаговая трассировка = **мозг защёлкивается**.

**Цена:** очередной раз интерактивное выполнение = не теория, а ощущаемый код.

### 5. **FSRS-Повторы на Основе Реальных Собес-Паттернов** (измерение обвязка)
**Идея:** трек не просто учит 6 механизмов, а **ловит** типичные завальки на собесах (из references/failures.md или живых записей) и строит очередь повторов на них. Пример:
- `is` vs `==` (раздел 1) → типичный завал (на записи 2026-07 спросили, пользователь завис 1.5 мин).
- `list` vs `set` vs `tuple` (раздел 3) → завал на разборе (назвал только mutable/immutable, забыл O(1) поиск в set).
- `@classmethod` vs `@staticmethod` (раздел 9) → завал на примере полиморфизма.
- `yield` vs `return` в фикстуре (раздел 7) → завал на cleanup.

Вместо обычного FSRS-расписания — **динамическая сложность**: вопрос, который ловит завал, автоматически получает короче интервал и более сложный follow-up (например, combo-вопрос: «что если поменять `yield` на `return` И убрать `finally`?»).

**Почему вау:** FSRS-тренажёр, обученный на реальных провалах, = уникальная траектория усвоения. Не обобщённый учебник, а **меднолись вашего мозга под конкретные подвохи собеса**.

**Цена:** повторение превращается из скучного в хирургическое, не-забываемое.

## Бонус-Идеи (Уровень 6–8)

**6. Context manager как Мульт:** интерактивная аннимация `__enter__/__exit__` с exception-handling (try/finally под капотом).  
**7. Async/Await как Стэнфорд-Лекция:** event loop, 3 параллельных `await` vis-à-vis 3 последовательных с `asyncio.gather()` (бок-о-бок кривые времени выполнения).  
**8. Pydantic-Валидация как Breaking-API:** введи невалидные данные в карточку → видишь ValidationError **в реальном времени** (не текстовый пример).
