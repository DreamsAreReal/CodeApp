# Design — lessons-corpus, волна 1 (инфраструктура + миграция + трек C# core, ~41 урок)

North Star: корпус глубоких анимированных уроков (книги + C# + инструменты) в существующем TMA с FSRS; каждый урок — уровень ниже абстракции источника.

## Архитектура и контракты

**Реестр треков/разделов (новое)** — `app/src/lessons/registry.ts`:
`TRACKS: Track[]` → `{ id: "CS", title, sections: Section[] }`; `Section = { id: "CS.S1", title, order, prereqs: SectionId[], lessons: LessonManifestEntry[] }`; `LessonManifestEntry = { id, slug, title, cards: number, load: () => Promise<LessonData> }` (dynamic `import()` — Vite режет чанк на урок автоматически). `index.ts` сохраняет экспорт списка метаданных для харнессов, но БЕЗ статических импортов тел уроков. `types.ts`: `track: string` (union расширяется: "CS" | "PY" | будущие "TOOLS"/"BOOKS-*"), `LessonLang` += `"none"` (инструменты/книги без языка), `LessonData.section: string` (обязательное новое поле).

**ID-схема волны 1**: `CS.S<раздел>.<slug>`, разделы: S1 Типовая система (10 ур.), S2 Async/Await (9), S7 Память и GC (10), S17 Коллекции-внутренности (7), S18 Итераторы/yield (4), S4 Делегаты (в волне 1 — только урок closures, `CS.S4.closures-capture`). Полный перечень тем и якорных URL — RS-03 v2 §2 (разделы 1, 2, 7, 17, 18, 4.3); реестр в features.md перечисляет слаги.

**Бэкенд**: сид — как раньше, файл на урок `backend/Codex.Backend/seed/lessons/<id>.json` (+поле `section`). **Лимит новых карт**: в выдаче `/api/due` never-reviewed карты ограничиваются `NEW_CARDS_PER_DAY` (конфиг, дефолт 10), порядок — по `Section.order` и порядку урока в разделе; review-карты (уже виденные) не ограничиваются. Расчёт «новых сегодня» — по счётчику новых карт, впервые выданных за календарный день (UTC, поле в user-стейте).

**Миграция удаления** (одна транзакция работ): удалить 6 TS-файлов старых C#-уроков + их сид-JSON; правка `index.ts`/registry; правка харнессов `verify/run.mjs` (boxing/c1, value-vs-reference/c1 → карты новых уроков), `verify/new-lessons.mjs` (gc/closures/async-await/hashtable → новые id); чистка dev-БД: DELETE из review_state/progress_events по осиротевшим lesson/card id (скрипт `backend/tools/purge-orphans.sql` или эндпойнт-миграция при старте). Py-трек не трогается.

**Авторская обвязка** (реализовано в F5): (1) анти-echo lint в `verify/density.mjs`: у карточки с `verify.run` ответ `expect` не должен ЭМИТИТЬСЯ командой run. **Уточнение реализации (F5)**: не наивная подстрока (у реальных карт `run` — команда-файл `python3.12 X_c3.py`/`dotnet run`, где короткий expect ложно матчит цифру в имени/интерпретаторе), а «expect внутри emit-вызова» (`echo`/`print`/`Console.Write`/…) — ловит именно echo-фальсификацию, компенсация G7-аудитом сохранена; (2) **`verify/density.mjs` (G7-скрипт)**: обход сцен всех уроков — доля сегментов с изменением узлов/рёбер/code-pointer между кадрами ≥90%, ≥4 сегмента и ≥2 карточки на урок, exit≠0. **Уточнение (F5)**: count-gate (≥4 сег/≥2 карты) применяется к АВТОРСКОМУ треку (CS+будущие); PY-трек grandfathered (неприкосновенен, G5; PY.M12/M13 легально по 3 сегмента) — динамика и анти-echo применяются ко всем; (2a) `verify/density-fixtures.mjs` — self-test lint-а (нарушитель ловится); (3) инкрементальный verify: `npm run verify:all -- --lessons <id,...>` = density scoped + fixtures без браузера (полный прогон 5 харнессов + density + fixtures — на границе майлстоуна); (4) `run-csharp` rate-limit 60 POST/мин: exec-ответы пре-замерены в `verify.expect`, обвязка run-csharp не дёргает; браузер-харнессы пейсят свои `/api/review`. **Внимание: текущий `npm run verify` запускает ТОЛЬКО run.mjs** (`app/package.json:13`) — до появления `verify:all` (F5) «полный прогон» = явное перечисление: `npm run verify && node verify/viz-fit.mjs && node verify/shell.mjs && node verify/new-lessons.mjs && node verify/loop.mjs`. **G7-аудит точности**: на каждом майлстоуне M3–M6 evaluator берёт случайную выборку ≥20% уроков майлстоуна (seed выбирает evaluator, фиксирует в вердикте), для каждого урока выборки — fetch источника и сверка ≥3 ключевых утверждений.

## Опыт/ценность (владелец смысла)
Первая сессия нового пользователя: home → трек «C#/.NET вглубь» с разделами и прогрессом → раздел S1 открыт первым (prereq-цепочка S1 → S7 → S17 → S18 → S2; мягкая рекомендация порядка, жёсткого lock в волне 1 нет) → урок 1 `CS.S1.type-system-map` → на ~3-й минуте первая машинная панель (память/IL под типами) — **первый «вау» ≤5 минут**; сегменты автоиграют, карточка exec с реальным stdout закрепляет. День N: due ≤25 карт, новых ≤10 — сессия 5–10 минут. Каждая фича ниже ссылается на этот путь.

## Данные
Уроки — TS-данные (LessonData) + сид-JSON; прогресс — SQLite (review_state, progress_events); реестр покрытия волны — features.md (таблица «раздел → уроки → статус»). Провенанс: sources[] каждого урока = URL Learn из RS-03 v2.

## Инструментарий верификации
`npm run build` (tsc+vite, чанки видны в выводе) · до F5: `npm run verify && node verify/viz-fit.mjs && node verify/shell.mjs && node verify/new-lessons.mjs && node verify/loop.mjs`; после F5: `npm run verify:all` (то же + density.mjs + анти-echo lint) · `dotnet test` · G-EXEC: `curl -X POST http://localhost:5080/api/authoring/run-csharp -d '{"code":"..."}'` · симуляция очереди: `node verify/sim-14d.mjs` (новый скрипт по образцу loop.mjs: 14 виртуальных дней, каждый — due → grade Good → следующий; ассерты: новых/день ≤10, due/день ≤25). **Реализация (F3, отступление зафиксировано)**: multi-day прогон требует единого управляемого времени для `/api/due` (дата гранта новых карт) И `/api/review` (FSRS-расписание); добавлен DEV-ONLY хедер `X-Sim-Now` (ISO-8601), гейтед `devMode` — переопределяет «now» запроса на симулированное; в Production игнорится, контракт `/api/review` при отсутствии хедера байт-в-байт цел. Семантика лимита — «≤10 ПЕРВЫХ грантов новых карт за UTC-день» (не «≤10 видимых»): review не рефилит слот того же дня, pending-карты не теряются; контр-флуд — метрика due≤25. · скриншоты: 375 (основной, TMA), 768 — `docs/tasks/lessons-corpus/evidence/F<N>/`.

## Турниры (фаза 4, черновики — одноразовые дешёвые агенты, интегрирует builder)
1. Нулевой экран: home-навигация трек→раздел (3 радикально разных варианта, парные сравнения).
2. Signature: шаблон «машинной панели» урока (3 варианта подачи: IL-лента / счётчики-замер / X-ray стек-куча).
