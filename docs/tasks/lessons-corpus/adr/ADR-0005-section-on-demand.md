# ADR-0005 — грузить тела уроков ПО РАЗДЕЛУ / по требованию (не все жадно)

## Статус
Принято 2026-07-18 (M2, долг evaluator-а R-05: prefetch подрывает G1)

## Контекст
ADR-0003 разнёс тела уроков на отдельные чанки (Vite code-split) — начальный чанк
упал 571.88 → 111.30 KB raw (G1 достигнут по РАЗМЕРУ бандла). Но `main.ts` после
показа home вызывает `void prefetchAll()`, который жадно грузит тела ВСЕХ уроков в
фоне. Замер (verify-харнесс, 2026-07-18): на home скачиваются **22 / 22** тел уроков.

При корпусе 100+ уроков это обнуляет смысл G1: entry-чанк мал, но сеть всё равно
тянет сотни чанков сразу после старта — в Telegram WebView (медленная догрузка,
single-origin) это тот же блокер, что и большой начальный бандл. G1-инвариант «начальный
чанк ≤ базлайн +10%» держится, но фактическая стоимость первой сессии деградирует
линейно с ростом каталога. Метаданные (заголовок/иконка/число карт) — уже в entry-чанке
(`LESSONS: LessonMeta[]`), поэтому home и progress рендерятся БЕЗ тел; тело нужно
только при открытии урока (`runLesson` → `loadLesson`, уже с async-fallback на промах кэша).

## Рассмотренные варианты
| Вариант | Плюсы | Минусы | Замер |
|---|---|---|---|
| A. Жадный `prefetchAll()` (сейчас) | открытие любого урока «мгновенно» (тёплый кэш) | грузит ВСЕ тела на home → линейный трафик, обнуляет G1 при масштабе | home = 22/22 тел (baseline) |
| B. Полный lazy — ничего не префетчить, грузить только по клику | 0 лишнего трафика; entry-чанк — всё, что тянется на старте | первое открытие урока = сетевая догрузка (~5–9 KB gzip) — заметный, но короткий спиннер | — |
| C. **Section-scoped prefetch**: тёплым держим только тела уроков ПЕРВОГО раздела рекомендованного трека (по prereq-порядку); остальные — lazy по клику | первая сессия (вход S1) мгновенна; трафик home ограничен размером ОДНОГО раздела, не всего каталога; масштабируется — рост корпуса не бьёт по home | нужен выбор «какой раздел греть» (детерминирован: первый раздел трека CS по order) | целевой замер приёмки: home ≤ размер первого раздела, НЕ 22/22 |

## Решение
Вариант **C** (section-scoped), с деградацией к B для всего, что вне тёплого раздела.

- `prefetchAll()` заменяется на `prefetchSection(sectionId)` (грузит тела уроков одного
  раздела) + хелпер `firstSectionId()` (первый раздел первого не-legacy трека по `order`).
- `main.ts` после home вызывает `void prefetchSection(firstSectionId())` — греет только
  вход первой сессии (design «Опыт»: S1 открыт первым). Legacy/PY/остальные разделы —
  lazy: `runLesson` при промахе кэша уже рисует loading-скелет и `await loadLesson(id)`.
- Опционально (не в этом ADR): при ОТКРЫТИИ урока догревать его раздел — «следующий урок
  того же раздела мгновенен». Оставлено на F4-навигацию (там появляется контекст раздела).

## Последствия
- Первая сессия (S1) не теряет мгновенность; трафик home ограничен ОДНИМ разделом.
- Открытие урока вне тёплого раздела = короткая догрузка чанка (~единицы KB gzip) с
  loading-скелетом (контракт `runLesson` уже это умеет — ADR-0003 fallback).
- Приёмка: verify-харнесс на home считает скачанные тела — их число = размер тёплого
  раздела (не 22/22); открытие «холодного» урока по-прежнему работает (E2E).
- Полный прогрев уступает место масштабируемости: это осознанный размен «−мгновенность
  редких переходов» на «−линейный трафик старта» (G1 по существу, не только по размеру).
