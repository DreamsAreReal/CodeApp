# ADR-0001 — принять MCQ-fallback для non-exec карточек (волна 1)

## Статус
Принято 2026-07-18

## Контекст
`Card.verify` обязателен типом (`app/src/lessons/types.ts:90`, поле без `?`); детерминированная проверка generation-ответа существует только для исполняемого кода (`/api/authoring/run-csharp`). Темы без исполняемой истины (Docker, архитектура, книги-советы) не могут дать честный `verify.expect`. Проба (`.spikes/probe/REPORT.md`) показала: echo-ответ в `run` — фальсификация гейта; легальная форма — `type: "compare"/"explain"` + `verify: {run:"", expect:""}` → MCQ-fallback движка.

## Рассмотренные варианты
| Вариант | Плюсы | Минусы | Замеры из спайка |
|---|---|---|---|
| A. MCQ-fallback (существующий движок) | 0 изменений движка; детерминирован | распознавание слабее припоминания (retrieval strength ниже) | проба: MCQ-карточка проходит tsc и легальна по types.ts (tsc exit 0, REPORT.md) |
| B. Новый verify-механизм (Scene-дифф/структурное само-объяснение) | generation-припоминание | новый тип карт + движок + харнесс; не нужен волне 1 (все 41 урок — exec) | нет спайка — по правилу спайка победить не может |

## Решение
Волна 1: все карточки exec (C# core). Non-exec = MCQ (вариант A) до отдельного ADR при заходе на треки инструментов/книг; тогда же спайк варианта B (predict-scene из RS-00 — кандидат).

## Последствия
- Цена дефолта MCQ: более слабое припоминание на non-exec темах и Гудхарт-риск «удобные MCQ»; компенсация — G7-аудит (плотность+точность) и контраст-пары.
- Волна 2+ обязана вернуться к этому ADR до массового авторинга non-exec уроков.
