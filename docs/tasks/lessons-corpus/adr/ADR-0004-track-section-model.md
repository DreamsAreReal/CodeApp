# ADR-0004 — новый трек "CS" с обязательным полем section (вместо реюза T1/T2)

## Статус
Принято 2026-07-18

## Контекст
Старые C#-уроки жили в треках `T1`/`T2` (union в `types.ts`: "T1" | "T2" | "PY") без понятия «раздел»; навигация по 41+ уроку требует иерархии трек→раздел (brief P0). Удаление старых уроков + новые id — как избежать коллизий прогресса в БД (review_state ключуется lesson/card id)?

## Рассмотренные варианты
| Вариант | Плюсы | Минусы | Замеры из спайка |
|---|---|---|---|
| A. Новый трек "CS", новые id `CS.S<n>.<slug>`, поле `section` обязательное | 0 коллизий со старыми id (сироты вычищаются миграцией F2); union расширяем; разделы — первый класс | правки types.ts + сид-схемы | RS-baseline v2: review_state ключуется id — новые id гарантированно не наследуют чужой прогресс (grep-доказательство в RS-01) |
| B. Реюз T1/T2 с новыми уроками | меньше правок типов | риск коллизий слагов/статистики со старым прогрессом; иерархии всё равно нет | — |

## Решение
Вариант A. `track: "CS" | "PY" | string-расширение` (будущие TOOLS/BOOKS-*), `LessonData.section: string` обязательное, `LessonLang` += `"none"`. Сид-JSON получает поле `section`.

## Последствия
- Py-трек не трогается (id PY.* неизменны) — гарантия G5.
- Навигация home группирует по track→section из registry.
- Принятые риски: харнессы, знающие старые id, правятся в F2 (в скоупе миграции по brief).
