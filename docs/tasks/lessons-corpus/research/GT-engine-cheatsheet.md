# GT · Шпаргалка движка авторинга уроков

Точная деконструкция ДВИЖКА (не пересказ) — для аудита плотности G7 (`density.mjs`) и
как эталон планки жанра. Источники истины (прочитаны целиком 2026-07-18):

- `app/src/lessons/types.ts` — схема `LessonData`/`Segment`/`Card` (тип-контракт урока).
- `app/src/engine/types.ts` — схема `Scene`/`DiagramNode`/`DiagramEdge`/`Zone` (движок).
- `app/src/engine/layout.ts` — auto-layout v2 (`layoutScene`, `at:{zone,row,col}`/`{in}`).
- `app/src/engine/render.ts` — `sizeNode`, `W_LADDER`, `KIND_H` (размеры узлов).
- `app/src/lessons/boxing.ts` — LOCKED-эталон плотности (C#, 7 разборов + IL).
- `app/src/lessons/py-generators.ts` — эталон (Python, 6 разборов + рентген кадра).
- `docs/AUTHORING-AI.md` — плейбук авторинга.

Провенанс: A (первичка — код репозитория + типы). Дата: 2026-07-18. Уверенность: высокая

### 1.4 `Card` (SRS-карточка; `types.ts:78-92`) — что значит `verify.run`/`verify.expect`

Все поля обязательны:

| Поле | Тип | Смысл |
|---|---|---|
| `id` | `string` | `"c1"`… — бэк-элемент повтора = `` `${lesson.id}/${card.id}` `` |
| `type` | `"predict-output"\|"find-the-bug"\|"compare"\|"explain"` | тип; оба эталона — только `predict-output` |
| `engagementLevel` | `string` | напр. `"responding"` |
| `question` | `string` | HTML-вопрос |

| `correctIndex` | `number` | индекс верного варианта (fallback) |
| `xp` | `number` | напр. 10 |
| `okText` / `noText` | `string` | HTML-фидбек «верно» / «мимо» |
| `verify` | `{ kind:"exec"; run; expect }` | детерминированный гейт (ниже) |
| `sourceRefs` | `string[]` | id источников карточки |

**`verify.run`** = команда запуска сниппета (напр. `"dotnet run"`, `"python3.12 PY.M6_c1.py"`).
**`verify.expect`** = ОЦЕНИВАЕМЫЙ ЭТАЛОННЫЙ ОТВЕТ — РЕАЛЬНЫЙ `stdout` исполнения (C#: Roslyn
через `POST /api/authoring/run-csharp`; Python: реальный прогон), НЕ «LLM думает».
Механика runner-а (AUTHORING §9): если `type==="predict-output"` И `expect` непустой →
рендерится ТЕКСТОВЫЙ ВВОД (пользователь ГЕНЕРИРУЕТ вывод, не выбирает); ответ сверяется
строкой с `expect` (нормализация: trim / CRLF→LF / срез хвостовых пробелов построчно) →
объективное ✓/✗, которое ПРЕДВЫБИРАЕТ FSRS-оценку (✓→Хорошо, ✗→Снова). `\n` в `expect`
= многострочный вывод (напр. `"created\nstart\n1\nmiddle\n2"`).
ГРАБЛИ (AUTHORING §9): run-csharp — TOP-LEVEL statements, НЕ оборачивать в `class/Main`.

### 1.5 `Source` (`types.ts:21-35`) — провенанс с заморозкой

`{ id, kind, org, title, url, date, archived? }`. `archived` = Wayback-снимок `url` на `date`
(живые MS Learn дрейфуют → снимок пинит точную формулировку дословных цитат). Соглашение

`«…»` — ЗАМОРОЖЕНЫ, расширять нельзя; помечены блоком `// COMPOSITE-QUOTES:` в файле урока.

## 2. Пять примитивов размещения + auto-layout v2

### 2.1 Пять `kind` узла (`engine/types.ts:23`, `NodeKind`)

`type NodeKind = "slot" | "ref" | "obj" | "chip" | "gate"`. Общие поля узла:
`{ id, kind, at?, name?, value?, typeTag?, accent?, good?, ghost?, state?, label?, detail? }`.

| kind | назначение | несущие поля | пример из эталона |
|---|---|---|---|
| `slot` | именованная ячейка (value-переменная / поле) | `name`, `value` | `{id:"i", kind:"slot", at:{zone:"stack",row:0}, name:"i", value:"123"}` |
| `ref` | ссылка (стрелка идёт от неё через `edge`) | `name` | `{id:"o", kind:"ref", at:{zone:"stack",row:1}, name:"o", accent:true}` |
| `obj` | объект на куче / генератор | `typeTag`, `value` | `{id:"obj", kind:"obj", at:{zone:"heap",row:0}, typeTag:"[int]", value:"123"}` |
| `chip` | лёгкая метка/копия/событие | `value` | `{id:"cp", kind:"chip", at:{zone:"heap",row:1}, value:"123"}` |
| `gate` | проверка типа / исход (✓/✗) | `state:"ok"\|"fail"`, `label`, `detail` | `{id:"gate", kind:"gate", at:{zone:"gate",row:0}, state:"fail", label:"short ≠ int", detail:"InvalidCastException"}` |

Модификаторы визуала: `accent:true` (акцентный/новый), `good:true` (успех), `ghost:true`
(призрак — узел, покидающий сцену / фон; в py-generators помечает исчезающие setup-чипы).

### 2.2 `Zone` (статическая подложка; `engine/types.ts:84-99`)

`{ id?, x, y, w, h, cls, label, labelCls, lx, ly, sub?, subCls?, subY? }`. `id` ОБЯЗАТЕЛЕН,
если на зону ссылаются узлы через `at:{zone}`. Зоны задаются координатами явно (это фон, не
контент); контентные узлы координат НЕ несут. Канон-ширина зон 138u, парой стек+куча при
x=14 и x=188. `cls` несёт семантику: `"vz-zone"` (стек), `"vz-zone heap"` (куча),
`"vz-zone good"` (успех), `"vz-zone xray"` (рентген кадра в py-generators).

### 2.3 auto-layout v2 — `at:{zone,row,col}` / `at:{in}` (`layout.ts`)

ПРИНЦИП (AUTHORING §3a): узел объявляет только МЕСТО, движок (`layoutScene`) считает
`x/y/w/h`. Кривой/вылезающий кадр СОБРАТЬ НЕЛЬЗЯ — либо валидный кадр, либо брошенная
ошибка авторинга (никогда тихий баг). Escape hatch — явные `x/y` без `at` (snap+clamp).

Два вида `at` (`NodePlacement`, `engine/types.ts:36`):

- **`at:{ zone, row, col? }`** — узел в сетке зоны. Узлы с одним `(zone,row)` = РЯД
  (слева-направо по `col`, затем по порядку в массиве). `col` отсутствует ⇒ col 0.
  Модель СЕТКИ (`layout.ts:188-209`): колонки = объединение всех `col`; ширина колонки =
  MAX ширина ячейки по рядам; ОДИН стабильный center-X на колонку (пустая ячейка колонку не
  сдвигает — так 2D-таймлайн держит «левые» узлы разных рядов на общем X). Высота ряда =
  MAX высота узла ряда; ряды стопкой сверху вниз, весь блок центрируется в зоне.
- **`at:{ in, order? }`** — узел ВЛОЖЕН в родителя (напр. value-поле внутри obj-класса).
  Родитель авто-растёт по высоте/ширине под детей ниже header-полосы (`layout.ts:104-120`).

Инварианты движка (`layout.ts:12-21`, проверяются `verify/viz-fit.mjs`): в зоне PAD≥8 со
всех сторон · один center-Y на ряд (высоты могут различаться) · нет overlap · grid-snap
(GRID=2) · ширины на лестнице. Переполнение колонок → сначала ужим ширин ВНИЗ по лестнице,
потом БРОСОК ошибки. Ряды выше зоны → бросок ошибки. Узел без `at` и без `x/y` → ошибка.

Токены (`layout.ts:26-33`): `PAD=8`, `GUTTER_CROSS=16` (между узлами ряда), `GUTTER_ROW=16`
(между рядами), `HEADER_H=20` (полоса под `typeTag`), `NEST_GAP=6`, `W_LADDER_MIN=56`.

### 2.4 Размеры узла — `sizeNode` (`render.ts:56-78`)

`w/h` авторить НЕ надо. Ширина: лестница из 6 ступеней `W_LADDER = [56, 72, 96, 120, 144,
168]` — движок мерит текст (`monoMeasure`) и берёт узкейшую ступень ≥ нужной; при переполнении
ужимает вниз по лестнице. Высота: одна на `kind` (`KIND_H`), с bump на «высокий вариант»
только когда текст требует — `gate` с `label`+`detail` (две строки) или `obj` с `typeTag`+
`value` (`render.ts:66`). Явный `w` (напр. `w:96`, `w:120`) допустим — снап+клэмп на лестницу.

## 3. Что делает сегмент «динамическим» — признаки для `density.mjs` (G7)

ПРИНЦИП: сегмент «живой», если МЕЖДУ соседними кадрами `scenes[]` МЕНЯЕТСЯ граф узлов/рёбер
— тогда движок производит переход (enter/update/exit + FLIP), а не показывает статичную
картинку. Один кадр или N идентичных кадров = НЕ анимация (провал плотности, AUTHORING §7:
«1 анимация + буллеты» — это провал). Диф-модель — `DiffResult {enter, update, exit}` по
`key`/`id` узла (`engine/types.ts:108-112`).

Механические признаки для скрипта (по паре соседних сцен `k`→`k+1` внутри сегмента):

1. **Δ множества узлов**: `id`, появившиеся (ENTER) или исчезнувшие (EXIT) между кадрами.
   Признак: `set(nodes[k+1].id) != set(nodes[k].id)`. Пример boxing s1: кадр0 `{i}` →
   кадр1 `{i, obj, cp}` (enter `obj`,`cp`) → кадр3 `{i, o, obj}` (exit `cp`, enter `o`).
2. **Δ множества рёбер**: `edges` появились/исчезли (напр. ребро `o→obj` возникает в
   действии 3). `set(edges[k+1].id) != set(edges[k].id)`.
3. **Δ размещения (FLIP-движение)**: у узла с тем же `id` сменился `at` (zone/row/col) —
   движок делает transform-переход. Пример boxing s1: `cp` переезжает `stack,row1` →
   `heap,row1` (копия «переносится» со стека в объект).
4. **Δ полей узла (UPDATE)**: у того же `id` сменились `value` / `state` / `accent` /
   `ghost` / `typeTag`. Примеры: s2 `i.value` `123`→`456`→`123`; s3 `gate.state`
   `ok`→`fail`; py-generators s2 `loc.value` `n=41`→`n=42`, `g.value` `paused`→`running`.
5. **Δ синхронной панели**: между кадрами меняется `codeLine` / `ilLine` / `out` (движется
   подсветка кода/IL или растёт консоль) — сопровождает граф-переход.

Критерий динамичности сегмента для G7: `scenes.length ≥ 2` И ≥1 из признаков 1–4
срабатывает хотя бы на одной паре соседних кадров. Практический порог эталонов: 2–4 кадра
на сегмент, и КАЖДЫЙ переход несёт содержательное Δ (не косметику). Сегмент с `scenes.length
== 1` или с идентичными по узлам/рёбрам/полям кадрами → флаг «статичный / провал плотности».

Дополнительный сигнал плотности (не про динамику, но про «показано, не упомянуто»): у
каждого сегмента непустой `explain` С дословной цитатой (эвристика: наличие `«…»`/`&laquo;`
и хотя бы одного `sources[]`), непустой `caption` у КАЖДОГО кадра.

## 4. Чек-лист «урок на планке boxing.ts»

Эталон `boxing.ts` (LOCKED density exemplar) + сверка с `py-generators.ts`:

**Счётчики (планка):**
- Сегментов (разборов): **7** (boxing) / 6 (py-generators). Плейбук §2: флагман ≥6,
  стандартный ≥2–3. Один нюанс/мисконцепция = ОДИН сегмент.
- Сцен на сегмент: **2–4** (boxing: s1=4, s2=3, s3=4, s4=3, s5=3, s6=3, s7=3). Каждый
  переход между сценами содержателен (см. §3).
- Карточек: **1** (boxing) / **4** (py-generators). Рекомендация плотности (AUTHORING §3.4):
  лесенка `predict → modify → explain`; py-generators показывает MODIFY-ступень (c4 = c2,
  «починенный» материализацией в список: `5 и 5` вместо `5 и 0`).
- `takeaways`: ровно **3** — по одному `why` / `cost` / `avoid`.
- `misconceptions`: ≥1 с HTML-`hook` (секция 1 = refutation мисконцепции сеньора).
- `spec` + `edgeCases`: непустые, каждый `Claim` со `source`.

**Как выглядит «машинная панель»:**
- Байткод-панель `il[]` (для C#): массив `{off, op, arg, cmt}`, где `off`=`"IL_0002"`,
  `op`=`"box"`, `arg`=`"int32"`, `cmt`=`"// 0x8C"` (реальные hex-опкоды: `box` 0x8C,
  `unbox.any` 0xA5 — из первоисточника, НЕ выдуманы). Синхронна с анимацией через
  `scene.ilLine`. Для Python аналог — dis/байткод-панель (badge переключает `lang`).
- Консоль-панель: `console:true` на сегменте + `scene.out` на кадрах (растёт построчно);
  используется в predict-разборах (s2 boxing, s1/s3/s4/s6 py-generators).
- Код-панель: `code[]` слева, активная строка подсвечена `scene.codeLine`.
- Gate-узел как «машинный вердикт»: `state:"ok"/"fail"` + `label` + `detail` (напр.
  `int == int ✓`, `InvalidCastException`, `StopIteration`, `NullReferenceException`).

**Тон (планка):**
- СЕНЬОР, без азов/сюсюканья, ставка «как будто я сам спроектировал технологию» — глубина
  на машинный уровень (IL, память, GC, замороженный кадр gi_frame), а НЕ написание кода.
- Глубина ВПЛЕТЕНА в поток (сегменты по местам), НЕ отдельный аккордеон «копнуть глубже».
- Каждый `explain` = «Механизм — как и почему» + ДОСЛОВНАЯ короткая цитата первоисточника в
  `«…»` (verbatim, verified word-for-word; числа/опкоды — только из исполнения/спеки).
- Expertise-reversal: меньше лесов, сразу глубже, edge-cases как first-class.
- `foot` подводит итог форматом: `урок · <b>X</b> · N анимир. разборов · … · дизайн <b>mid</b>`.
- Дизайн-токены mid НЕ отклонять (AUTHORING §5): cream/coral/sage/amber, Rubik+Onest+JetBrains
  Mono, line-SVG иконки (НЕ эмодзи), анти-слоп (без золота/тёмного-кино/Inter).

**Провалы планки (флаги для G7, AUTHORING §7):**
- «1 анимация + буллеты с нюансами» вместо сегмента-на-нюанс.
