# AUTHORING-AI — как ИИ добавляет урок (playbook)

Читай ЭТО целиком перед созданием урока. Цель — чтобы ты (другой ИИ / будущая сессия) мог добавить
урок НА ПЛАНКЕ без истории чата. Продукт: Telegram Mini App для СЕНЬОР-разработчика — «пройтись по
базе разработки заново, но ГЛУБОКО». Ставка: **«как будто я сам спроектировал технологию»** — глубина
понимания (машинный уровень, IL, внутренности), а НЕ написание кода в аппе.

## 0. Золотой эталон и артефакты-истины
- **Эталон урока (копируй планку отсюда):** `docs/research/concepts/lesson-boxing/index.html` — 7
  анимированных разборов, каждый = анимация + «Механизм — как и почему» + дословный первоисточник.
- **Формат данных:** `docs/design/lesson-format.md` (lesson-as-data + `segments[]`).
- **Дизайн-токены (НЕ отклоняться):** `docs/research/concepts/mid/index.html` + память
  `design-locked-cream-coral-rubik`.
- **Методика (наука):** `docs/research/RS-09-instructional-design-depth.md` (7-фазный шаблон,
  expertise-reversal, refutation мисконцепций).
- **Точность/пайплайн:** `docs/research/RS-10-content-pipeline-schema.md` (детерминированные гейты).
- **Движок анимаций:** `app/src/engine/` (после сборки скелета) — Scene→SVG→data-join→FLIP→StepPlayer.
- **Бэкенд/API:** `backend/README.md` (auth/due/review; авторский `run-csharp` = проверить ответ реальным `dotnet`).

## 1. Ментальная модель
Урок = **ДАННЫЕ** (объект `lesson` с `segments[]`). Движок и дизайн — общие. Добавить урок =
добавить файл данных в `app/src/lessons/` + строку в реестр. НЕ верстай урок вручную, НЕ трогай
движок/дизайн. Один объект `lesson` драйвит: анимации (`viz.scenes`), карточки SRS (`cards`), текст.

## 2. ПЛАНКА ПЛОТНОСТИ (главное — не нарушать)
- Урок = МНОГО анимированных РАЗБОРОВ, каждый нюанс ПОКАЗАН анимацией и ОБЪЯСНЁН («как и почему»),
  а НЕ «упомянут строкой». Флагман ≥6 сегментов, стандартный ≥2–3.
- Тон — СЕНЬОР: без азов, бей в нюансы/машинный уровень/gotchas/«почему так спроектировано». Expertise-
  reversal (RS-09): меньше лесов, сразу глубже, edge-cases как первый класс.
- Иди в ГЛУБИНУ на тему (IL/память/GC/стейт-машина/веса), а не в ширину.

## 3. Рецепт «добавить урок» (по шагам)
1. **Выбери концепт** из C#-ядра (или другого трека) и перечисли его НЕСУЩИЕ нюансы/мисконцепции
   (то, что сеньор «вроде знает», но хочет прибить). Пример boxing: механика упаковки, копия≠ссылка,
   гейт распаковки+исключения, IL box/unbox.any, где живёт значение, цена в цикле/GC, generics+скрытый boxing.
2. **Найди первоисточник на КАЖДОЕ утверждение** (learn.microsoft.com для C#/.NET — используй MCP
   Microsoft Learn; ECMA-334/335 для спеки/IL; arXiv для LLM; CLRS/DDIA для CS/БД). Возьми ДОСЛОВНЫЕ
   короткие цитаты. Источник — ОБЯЗАТЕЛЬНОЕ поле каждого claim (без него урок не годен).
3. **Составь `lesson` (segments[])** — на каждый нюанс сегмент: `{id, kicker, title, code?, il?,
   viz:{viewBox,zones,scenes[],predictAt?}, explain, sources[]}`. Собери `viz.scenes` из ПРИМИТИВОВ (§4).
4. **Карточки `cards[]`** — активное припоминание (СИЛЬНОЕ = генерация, не узнавание):
   - `predict-output` с непустым `verify.expect` → рендерится как **ТЕКСТОВЫЙ ВВОД** («напечатай
     вывод»): пользователь ГЕНЕРИРУЕТ ответ, а не выбирает из вариантов. Ответ сверяется строкой
     с `verify.expect` (толерантная нормализация: trim, CRLF→LF, срез хвостовых пробелов построчно)
     → ОБЪЕКТИВНОЕ верно/мимо. `verify.expect` — это **ОЦЕНИВАЕМЫЙ ЭТАЛОННЫЙ ОТВЕТ** (реальный
     stdout `dotnet` через `POST /api/authoring/run-csharp` — НЕ «LLM думает»), а `options`/
     `correctIndex` теперь **fallback/подсказка** (используются, только если у карточки нет годного
     `expect`, напр. `compare`/`explain`).
   - `compare`/`explain` (структурированное само-объяснение по ключевым пунктам — НЕ LLM-судья как
     оценка, см. RS-14) без исполнимого `expect` → остаются на MCQ-fallback.
   - Объективный результат **предвыбирает FSRS-оценку** (верно → Хорошо, мимо → Снова); само-оценка
     теперь вторична. Мимо → карточка возвращается в этой же сессии (FSRS-6 learning step ~1 мин).
   - Рекомендация плотности: НЕСКОЛЬКО карточек на нюанс — лесенка **predict → modify → explain**
     (предскажи вывод → измени код под цель → объясни механизм), чтобы отслеживать понимание, а не
     угадывание одного варианта. `verify` детерминирован; для generation — обязателен непустой `expect`.
5. **Проверь точность (гейты, RISK №1):** каждый идентификатор существует (NuGet/`dotnet build`); ответ
   карточки = реальный вывод исполнения; цитаты — дословно из источника. НЕ выдумывай числа/опкоды.
6. **Зарегистрируй** урок в `app/src/lessons/index.ts` и, при необходимости, в сид бэка/`/api/lessons`.
7. **Верифицируй** (§6) headless-прогоном.

## 3a. РАЗМЕЩЕНИЕ виз-узлов — ТОЛЬКО через `at` (auto-layout v2, bug-proof)
**Правило:** узел НЕ несёт координат `x/y` (и почти никогда `w/h`). Он объявляет только
МЕСТО, а движок (`layoutScene`) сам считает позицию, размер, выравнивание и маршрут рёбер.
Кривой/вылезающий/несбалансированный кадр СОБРАТЬ НЕЛЬЗЯ — это гарантия движка, проверяемая
`verify/viz-fit.mjs` (AUTHORING-PROOF: in-zone PAD≥8 · один center-Y на ряд · нет overlap ·
grid-snap · ширины на лестнице). Declare placement — the engine positions.

- **В зоне:** `at: { zone: <zoneId>, row: <int>, col?: <int> }`. Дай каждой зоне `id`.
  Узлы с одинаковым `(zone,row)` образуют РЯД (слева-направо по `col`, затем по порядку в
  массиве); ряд центрируется горизонтально, все узлы ряда делят ОДИН center-Y (высоты могут
  различаться — chip и obj в одном ряду выравниваются по центру). Ряды стопкой сверху вниз,
  весь блок центрируется в зоне.
- **Вложенность:** `at: { in: <parentId>, order?: <int> }` — узел кладётся ВНУТРЬ другого
  (напр. поле display-класса внутри его obj). Родитель авто-растёт по высоте, чтобы вместить
  детей под своей «шапкой» (typeTag). Ширину/позицию вложенного считает движок.
- **Размер:** приходит из `sizeNode` (высота по kind + ширина по лестнице под измеренный
  текст). `w/h` в узле писать НЕ надо — движок подберёт и при переполнении зоны ужмёт
  ширину до подходящей ступени лестницы, а если не влезает — БРОСИТ ошибку авторинга
  (виден сбой, не тихий баг). Узел без `at` и без `x/y` → тоже ошибка авторинга.
- **Escape hatch:** явные `x/y` (без `at`) ещё уважаются (snap+clamp в viewBox) для редких
  спецслучаев, но ДОКУМЕНТИРОВАННЫЙ путь — только `at`. Не мигрированные уроки на `x/y`
  продолжают работать без изменений (обратная совместимость).

**Было (ручные координаты — так БОЛЬШЕ НЕ надо):**
```ts
nodes: [
  { id: "n",  kind: "slot", x: 83,  y: 88,  name: "n", value: "1" },
  { id: "add", kind: "ref", x: 83,  y: 152, name: "add" },
  { id: "cap", kind: "chip", x: 257, y: 112, value: "→ n" },
]
```
**Стало (только размещение — движок расставит):**
```ts
// zones: [{ id:"stack", ... }, { id:"heap", ... }]
nodes: [
  { id: "n",   kind: "slot", at: { zone: "stack", row: 0 }, name: "n", value: "1" },
  { id: "add", kind: "ref",  at: { zone: "stack", row: 1 }, name: "add" },
  { id: "cap", kind: "chip", at: { zone: "heap",  row: 0 }, value: "→ n" },
]
```
Вложение (поле `nf` внутри display-класса `dc`, `dc` авто-растёт):
```ts
{ id: "dc", kind: "obj",  at: { zone: "heap", row: 0 }, typeTag: "DisplayClass" },
{ id: "nf", kind: "slot", at: { in: "dc" }, name: "n", value: "1" },
```
Эталон полной миграции — `app/src/lessons/closures.ts` (5 сегментов: ряды, вложение, рёбра).

## 4. Каталог примитивов виз (собирай сцены из них; см. движок)
`stack{thread}` (кадр стека, подпиши «стек потока ~1МБ Windows» где уместно) · `heap` («GC-куча · общая ·
все потоки») · `thread{name}` · `box{name,value,type}` · `ref{from→to}` (стрелка) · `gate{ok/fail}`
(проверка типа) · IL-панель (`scene.ilLine` синхронна с шагом) · `buckets{n}`+`chain` (хеш) ·
`timeline{tracks,events}` (async/машина состояний) · `attn{tokens,weights}` (веса внимания) ·
`pointer/label/highlight`. Сцена = снимок из примитивов; движок диффит (enter/update/exit) + FLIP.
Нужен ПРИНЦИПИАЛЬНО новый тип виз → добавь ОДИН примитив в `app/src/engine/` (типизированно) — он
станет переиспользуемым во всех уроках. Подложка SVG; Canvas — только для «густых» сцен.

## 5. Дизайн (mid) — НЕ отклоняться
Токены: `--bg #F6F1E9`, `--surface #fff`, `--ink #23201B`, `--muted #8A8377`, `--line #E7DFD2`,
`--coral #F0533A`/`--coral-d #C43D28`, `--sage #7C8B6F`, `--amber #E8A13A` (чуть). Шрифты Rubik(700/800)
+ Onest + JetBrains Mono. Радиусы 14/18/22, мягкие тёплые тени. Иконки — line-SVG (НЕ эмодзи).
**Анти-слоп:** без золота/тёмного-кино/детсад-зелёного/Inter; без «упоминаний вместо анимаций»; без
streak-shaming; читаемость AA.

## 6. Автозапуск и верификация
- **Автозапуск:** каждая виз-сцена стартует при попадании в вьюпорт (IntersectionObserver), `prefers-
  reduced-motion` → финальный кадр статикой. Ручные ◀▶/play/скраббер сохраняются.
- **Верифицируй headless-Chrome (puppeteer-core есть в scratchpad):** (a) сегменты рендерятся, анимация
  играет (шаги/FLIP, автозапуск); (b) predict-гейт блокирует до раскрытия; (c) карточка шлёт `/api/review`
  и расписание меняется; (d) 0 ошибок консоли; (e) reduced-motion; (f) скролл. GIF прогона — для показа.
- **Аудит точности (G2):** независимая сверка выборки утверждений с источником; 0 несуществующих
  идентификаторов; ответы = реальное исполнение. Не ставь урок `verified`, пока не зелено.

## 7. Как НЕ надо (частые ошибки)
- НЕ «1 анимация + буллеты с нюансами» — это провал плотности (это чинили: см. историю boxing).
- НЕ отдельный аккордеон «копнуть глубже» — глубина ВПЛЕТЕНА в поток (сегменты по местам).
- НЕ азы/сюсюканье — аудитория сеньор.
- НЕ LLM-судья как оценка ответа (bias, RS-14) — код→скрытые тесты/exec; проза→структ. само-объяснение.
- НЕ выдумывай факты/цитаты/опкоды/числа — только из первоисточника, дословно, с ссылкой.
- НЕ меняй дизайн-токены/движок ради одного урока — расширяй общими примитивами.

## 8. Минимальный шаблон `lesson` (заполни)
```ts
export const lesson = {
  id: "T1.M?.slug", track: "T1", title: "…", audience: "senior", depth: 3,
  sources: [{ id: "ms-…", kind: "doc", url: "https://learn.microsoft.com/…", date: "YYYY-MM-DD" }],
  segments: [
    { id: "s1", kicker: "…", title: "…",
      code: ["…C#…"], il: [/* {off,op,arg,cmt} если уместно */],
      viz: { viewBox: "0 0 340 232", zones: [/* stack/heap/… */],
             scenes: [ /* снимки из примитивов; предскажи-then-run через predictAt */ ], predictAt: null },
      explain: "Механизм — как и почему (2–4 предложения, по-взрослому).",
      sources: ["ms-…"] },
    // …ещё сегменты: КАЖДЫЙ нюанс = свой…
  ],
  cards: [
    { id: "c1", type: "predict-output", verify: { kind: "exec", run: "dotnet run", expect: "<реальный вывод>" }, sourceRefs: ["ms-…"] }
  ],
};
```
Держи глоссарий: стек/куча/упаковка/мастерство; UI-текст RU, код/IL/термины EN.

## 9. Актуальная реализация в app/ (схема КАК ПОСТРОЕНА — сверяйся с ней)

Скелет собран. Ниже — точная схема из кода (`app/src/lessons/types.ts`). Отличие от
§8: в реализации поля виз (`viewBox/zones/scenes`) лежат ПЛОСКО в объекте сегмента
(без обёртки `viz:{}`), а `type LessonData` требует ещё `kicker/home/foot`.
Тип — источник правды; примеры-эталоны: `app/src/lessons/boxing.ts` (7 сегментов, IL)
и `app/src/lessons/value-vs-reference.ts` (4 сегмента).

**Файлы, которые ты трогаешь (и только их):**
- `app/src/lessons/<slug>.ts` — новый `LessonData` (богатый контент, драйвит анимации);
- `app/src/lessons/index.ts` — импорт + строка в `LESSONS` (порядок = concept-DAG);
- `backend/Codex.Backend/seed/lessons/<id>.json` — seed каталога повторов (те же `id` и `card.id`).

**Связь петли:** элемент повтора на бэке = `` `${lesson.id}/${card.id}` `` (напр.
`T1.M3.boxing/c1`). `id` урока и `id` карточек ОБЯЗАНЫ совпадать на фронте и в seed.

**Плоская схема сегмента (как в коде):**
```ts
{ id:"s1", num:"01", kicker:"Категория · подпись", title:"Заголовок разбора",
  viewBox:"0 0 340 210", zones: MM_ZONES,           // плоско, не под viz:{}
  code?:["int i = 123;"], il?:[{off,op,arg,cmt}],
  predictAt?:2, predictQ?:"HTML", console?:true,
  scenes:[ { codeLine?:0, ilLine?:1, out?:"123", caption:"HTML",
             // РАЗМЕЩЕНИЕ через at (auto-layout v2, см. §3a) — НЕ x/y. w/h тоже не нужны.
             nodes:[{id,kind:"slot|ref|obj|chip|gate",
                     at:{zone,row,col?} | {in,order?},   // предпочтительно; x/y = escape hatch
                     name?,value?,typeTag?,accent?,good?,state?,label?,detail?}],
             edges:[{id,from,to,accent?}] } ],
  explain:"HTML: механизм + ДОСЛОВНАЯ цитата", sources:["ms-x"] }
```

**Карточка** (`cards[0]` рендерится в скелете, замыкает FSRS-петлю):
```ts
{ id:"c1", type:"predict-output", engagementLevel:"responding",
  question:"HTML", options:["A","B","C","D"], correctIndex:0, xp:10,   // options = fallback/подсказка
  okText:"HTML", noText:"HTML",
  verify:{ kind:"exec", run:"dotnet run", expect:"РЕАЛЬНЫЙ вывод" }, sourceRefs:["ms-x"] }
```
**Механика ответа (runner):** если `type==="predict-output"` И `verify.expect` непустой → рендерится
ТЕКСТОВЫЙ ВВОД (`#qTyped`, монопространство), ответ сверяется строкой с `expect` (нормализация:
trim/CRLF→LF/срез хвостовых пробелов) → объективное ✓/✗; на мимо показывается diff «твой ↔ ожидалось».
Иначе (нет годного `expect`) → MCQ-fallback по `options`/`correctIndex`. Результат ПРЕДВЫБИРАЕТ
FSRS-оценку (✓→Хорошо, ✗→Снова); пользователь может уточнить (Трудно/Легко), но объективный сигнал —
главный. Перед раскрытием — лёгкий сигнал «уверен?» (да/нет), после — калибровочная реплика
(верно+уверен=отлично · верно+не уверен=«знал, но сомневался» · мимо+уверен=«переоценил» ·
мимо+не уверен=ожидаемо). `correct`+`confidence` шлются опциональными полями в `POST /api/review`
(default null — старый контракт цел) и копятся в стат «Калибровка» на экране Прогресс.
Debug-хуки для headless: `window.__lastAnswer` (typed/expected/correct/confidence),
`window.__calibration`, `window.__lastReview`.

**Гейт точности G-EXEC (обязателен перед сдачей)** — ответ = реальный вывод:
```bash
curl -s -X POST http://localhost:5080/api/authoring/run-csharp \
  -H 'Content-Type: application/json' \
  -d '{"code":"using System; using System.Collections.Generic; var d = new Dictionary<string,int>(); d[\"cat\"]=1; d[\"cat\"]=2; Console.WriteLine(d.Count);"}'
# stdout.trim() ДОЛЖЕН == expectedOutput/expect
```
ВАЖНО (грабли, стоившие часа): endpoint исполняет код как Roslyn **CSharpScript = TOP-LEVEL
statements**. НЕ оборачивай в `class P { static void Main(){…} }` — такой код КОМПИЛИРУЕТСЯ, но
`Main` не вызывается → пустой stdout, и ты решишь, что «ничего не печатает». Шли `using`-и
сверху + операторы верхнего уровня (как в примере). Ответ карточки = этот реальный stdout.trim().

**Проверка сборки:** `cd app && npm run build && npm run verify` (headless: урок
рендерится, анимация играет, карточка двигает расписание, 0 ошибок консоли).
Подробный README запуска — `app/README.md`.
